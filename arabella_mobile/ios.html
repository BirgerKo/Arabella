Arabella iOS

iOS-specific architecture & implementation — updated 2026-05-11 20:34

Overview

This document covers everything specific to the iOS target of Arabella Mobile. For the shared protocol library, FFI bridge, and overall architecture see Architecture & Protocol.

The iOS app is a native Qt/QML application with Cupertino-style controls. It communicates directly with Blauberg Vento fans over UDP on the local WiFi network using the same Rust protocol library that all Arabella platforms share. No cloud service or home-server relay is required.

Protocol library

Rust staticlib — shared with Android and SailfishOS. CMake preset and Rust target triples configured. Ready

C++ service & ViewModel layers

DeviceService, DiscoveryService, ViewModels. Not yet implemented. Phase 2

QML UI

Dashboard, Connect, Schedule, Scenario, Details pages. Not yet implemented. Phase 2

Platform files

Info.plist, entitlements, app icon, launch screen. Not yet created. Phase 2

Build Requirements

iOS builds must be performed on macOS. Cross-compilation from Linux or Windows is not supported by Apple tooling. The protocol library itself can be developed and tested on Linux (as in Phase 1), but the full iOS app requires a Mac.

  • Host OS macOS 14 (Sonoma) or later recommended
  • Xcode Xcode 15 or later; iOS 16+ SDK included
  • Qt Qt 6.6 or later with the iOS add-on installed via Qt Maintenance Tool
  • CMake 3.20 or later (bundled with Qt or installed separately)
  • Rust toolchain Stable Rust with aarch64-apple-ios and aarch64-apple-ios-sim targets
  • Apple Developer Active Apple Developer Program membership for device deployment and distribution
  • Provisioning Development provisioning profile covering the app bundle ID and the com.apple.security.network.client entitlement

Installing Rust iOS targets

rustup target add aarch64-apple-ios
rustup target add aarch64-apple-ios-sim

iPhone device requirements

The following steps must be completed on the target iPhone before Xcode can install a development build:

  • iOS version iOS 16.0 or later (set by MinimumOSVersion in Info.plist)
  • Developer Mode Enabled under Settings → Privacy & Security → Developer Mode. Required since iOS 16; the device must have been connected to Xcode at least once to surface this toggle.
  • UDID registered The device UDID must be added to the Apple Developer portal (Certificates, Identifiers & Profiles → Devices) before the provisioning profile is generated. Xcode can do this automatically via Window → Devices and Simulators → Add Device when signed in to a Developer account.
A free Apple ID (no paid membership) is sufficient for side-loading to a personally-owned device via Xcode. The paid Apple Developer Program ($99/year) is required for distribution to other users' devices, TestFlight, or App Store submission.

CMake Configuration

The ios preset in CMakePresets.json configures the build for iOS. It uses the Xcode generator so that CMake produces an .xcodeproj that Xcode signs and packages into an IPA. 

cmake --preset ios
open build/ios/ArabellaMobile.xcodeproj

The preset sets:

CMake variableValuePurpose
CMAKE_SYSTEM_NAMEiOSTells CMake to cross-compile for iOS
CMAKE_OSX_DEPLOYMENT_TARGET16.0Minimum iOS version
GeneratorXcodeRequired for iOS signing and IPA packaging

Corrosion + Xcode integration

Corrosion (v0.5) integrates the Rust crate into the Xcode CMake build. When CMake configures for iOS it instructs Corrosion to build the Rust crate for aarch64-apple-ios. The resulting libarabella_protocol.a staticlib is linked into the C++ service target automatically via target_link_libraries.

Corrosion and the Xcode generator require that CMAKE_SYSTEM_NAME=iOS is set before FetchContent_MakeAvailable(Corrosion) runs. The preset handles this correctly, but it must not be overridden manually on the command line after the fact.

Simulator vs device builds

The Simulator runs on the Mac's own CPU (Apple Silicon = arm64), so it uses a different Rust target than the physical device:

TargetRust tripleNotes
Physical iPhone/iPadaarch64-apple-iosRequired for real fan UDP testing
Simulator (Apple Silicon Mac)aarch64-apple-ios-simDifferent ABI from device despite same CPU arch
Simulator (Intel Mac)x86_64-apple-iosOnly needed for Intel Mac development
UDP testing against a real fan must be done on a physical device. The iOS Simulator uses a different network stack and does not reflect the permission dialog or the UDP multicast behaviour a user will actually experience.

Rust Cross-Compilation

The arabella_protocol Rust crate cross-compiles to iOS with no code changes. It uses only std::net::UdpSocket and std::ffi — both fully supported on Apple platforms.

XCFramework consideration

When distributing the library (e.g. as a Swift Package or for use in a future Swift/Objective-C layer), the two staticlibs should be combined into an XCFramework so Xcode can select the correct slice at link time: 

# Build both slices
cargo build --target aarch64-apple-ios --release
cargo build --target aarch64-apple-ios-sim --release

# Package as XCFramework
xcodebuild -create-xcframework \
  -library target/aarch64-apple-ios/release/libarabella_protocol.a \
  -library target/aarch64-apple-ios-sim/release/libarabella_protocol.a \
  -output ArabellProtocol.xcframework

For the CMake/Qt build path managed by Corrosion this step is not needed — Corrosion handles slice selection automatically through the Xcode build system.

Network Permissions (iOS 14+)

Starting with iOS 14, apps that send UDP packets on the local network must declare intent in Info.plist and receive a one-time user permission prompt before the first broadcast packet is allowed. This affects fan discovery only.

Required Info.plist keys

<key>NSLocalNetworkUsageDescription</key>
<string>Arabella uses the local network to discover and control
Blauberg Vento fans on your Wi-Fi network.</string>

<!-- Required when using Bonjour/multicast service types -->
<key>NSBonjourServices</key>
<array>
    <string>_arabella._udp</string>
</array>

Required entitlement

<key>com.apple.security.network.client</key>
<true/>

Manual connect as first-class path

If the user denies the local network permission, discovery is unavailable but the app remains fully functional. Manual connection (entering IP address, device ID, and password directly) bypasses discovery entirely and must be treated as a first-class UI path, not a fallback.

Connection methodRequires local network permissionWorks after denial
Auto-discovery (UDP broadcast)YesNo
Manual connect (IP + ID + password)NoYes

UDP port

The Blauberg Vento protocol uses UDP port 4000. This is an outbound connection from the app to the fan; no inbound listening port is required. The com.apple.security.network.client entitlement covers this use case.

UI Architecture

Style: Cupertino

Qt Quick Controls 2 ships with a Cupertino style that matches iOS system controls. It is activated by placing a qtquickcontrols2.conf file in the app bundle and importing QtQuick.Controls in QML — no code changes are needed to switch styles between platforms. 

[Controls]
Style=Cupertino

Page structure

PagePrimary content
DashboardPower button, speed control, mode selector, boost status, fan RPM, quick scenario slots
ConnectDiscovery list, connection history, manual entry form
Schedule8 day-groups × 4 periods each; enable/disable weekly schedule toggle
ScenariosSave current state as named scenario; apply, rename, delete; up to 4 quick-access slots on Dashboard
DetailsBoost delay, humidity threshold, current humidity, fan RPM, filter status, RTC sync, firmware version, alarm reset

Navigation model

iOS uses a TabBar (bottom tab bar, Cupertino style) as the root navigator. The Schedule and Details pages open as modal sheets from the Dashboard tab to avoid burying infrequently-used controls in the tab bar.

QML component reuse across platforms

Core QML components (PowerButton.qml, SpeedControl.qml, ModeSelector.qml, RpmDisplay.qml, HumidityWidget.qml) are written against Qt Quick Controls 2 and are platform-neutral. iOS-specific behaviour (Cupertino visual variants, iOS gesture handling) is handled by the style engine, not by branching QML code.

Polling

A QTimer in DeviceService calls vento_get_state() via the FFI bridge every 2 seconds while the app is in the foreground. Polling is suspended when Qt.application.state transitions to Qt.ApplicationSuspended or Qt.ApplicationInactive.

Platform Files

These files live under platform/ios/ and are specific to the iOS build. None have been created yet; they are required before Phase 2 can produce a runnable IPA.

  • platform/ios/Info.plist App metadata, bundle ID, display name, NSLocalNetworkUsageDescription, NSBonjourServices, minimum iOS version declaration. Required by Xcode; merged into the app bundle at build time.
  • platform/ios/Arabella.entitlements Declares com.apple.security.network.client (outbound network access) and com.apple.security.app-sandbox. Referenced in the Xcode project signing settings.
  • platform/ios/qtquickcontrols2.conf Sets Style=Cupertino for Qt Quick Controls 2. Placed in the app bundle resources so Qt picks it up at startup.
  • platform/ios/Assets.xcassets App icon set (1024×1024 base, Xcode generates all required sizes) and launch image. Required for App Store submission.
  • platform/ios/LaunchScreen.storyboard iOS launch screen displayed by the OS while the app initialises. Required by Apple App Store guidelines.

Bundle identifier

The bundle ID should follow reverse-DNS notation and be registered in the Apple Developer portal before the provisioning profile is created. Suggested: no.volue.arabella.

C++ Service Layer Interface

The service layer sits between the QML/ViewModel layer and the Rust FFI. It owns the fan connection, drives the poll timer, and dispatches all commands off the main thread.

Thread model

DeviceService lives on the Qt main thread. All blocking FFI calls (vento_get_state, vento_turn_on, etc.) run inside DeviceWorker on a dedicated QThread. Communication across the thread boundary uses queued signal/slot connections — the worker never touches main-thread objects directly.

The poll timer fires on the main thread and emits _sigPoll() to the worker via a queued connection. After every command the service also emits _sigPoll() immediately so the UI reflects the result without waiting for the next scheduled poll interval.

DiscoveryService

DiscoveryService runs a single UDP broadcast scan using QtConcurrent::run so the main thread stays responsive. It wraps vento_discover("255.255.255.255", 4000, …).

Signal / slotDescription

discoveryError(QString) signalEmitted if the FFI call returns an error status
isRunning()True while scan is in progress; drives the discovering property on DeviceViewModel

DiscoveredDeviceInfo carries: ip, deviceId, unitType, unitTypeName.

Scenario File Format (v2)

Scenarios are persisted to QStandardPaths::AppDataLocation + "/scenarios.json". The v2 format is identical to the Python desktop app (ventocontrol), so scenario files can be shared between platforms. The store caps at 10 scenarios and 3 quick slots per device.

v2 file structure

{
  "version": 2,

  "scenarios": [
    {
      "name": "Night",
      "fans": [
        {
          "device_id": "VENT-SIM-001",
          "settings": {
            "power": false,
            "speed": 1,
            "operation_mode": 0,
            "humidity_threshold": 65
          }
        }
      ]
    },
    {
      "name": "Day",
      "fans": [
        {
          "device_id": "VENT-SIM-001",
          "settings": {
            "power": true,
            "speed": 2
          }
        },
        {
          "device_id": "VENT-SIM-002",
          "settings": {
            "power": true,
            "speed": 3,
            "operation_mode": 2
          }
        }
      ]
    }
  ],

  "quick_slots": {
    "VENT-SIM-001": ["Night", "Day", ""],
    "VENT-SIM-002": ["Day", "", ""]
  }
}

Settings fields

All settings fields are optional — only captured fields are written. Missing fields are not applied when a scenario is activated.

JSON keyTypeDescription
powerboolFan on/off
speedintPreset speed 1–3; 255 = manual
manual_speedintManual RPM 0–255 (written together with speed: 255)
operation_modeint0 ventilation · 1 heat recovery · 2 supply
boost_activeboolBoost on/off
humidity_sensorint0 off · 1 on · 2 auto
humidity_thresholdintTarget %RH (40–80)

v1 → v2 migration

The v1 format stored scenarios per-device. ScenarioStore detects "version": 1 on load and migrates automatically — no user action required. The migrated data is not written back to disk until the next save operation. 

{
  "version": 1,
  "devices": {
    "VENT-SIM-001": {
      "scenarios": [
        { "name": "OldScenario", "settings": { "speed": 2 } }
      ],
      "quick_slots": ["OldScenario", null, null]
    }
  }
}

Quick slots

Quick slots are stored per-device under the quick_slots key as a 3-element array of scenario names. An empty string means the slot is unassigned. When a scenario is deleted, all quick-slot references to it are automatically cleared.

Eviction policy

When the 10-scenario cap is reached, the oldest scenario (index 0 of the internal list) is evicted before the new one is appended. Overwriting a scenario by name does not count as a new entry — it replaces in place.
