Arabella Mobile

Architecture, implementation & test reference — updated 2026-05-11 16:30

Introduction

Arabella Mobile is a cross-platform app for controlling Blauberg Vento heat-recovery fans. It targets iOS first, with Android and SailfishOS to follow, and aims for full feature parity with the existing ventocontrol PyQt6 desktop application.

The app communicates directly with fans over the Blauberg Vento UDP binary protocol (port 4000). No cloud service or home server is required; the user's phone connects to the fan on the local WiFi network.

Phase 1 — Protocol Library

Rust crate implementing the full Blauberg Vento protocol. Compiles to a static library and exposes a C ABI consumed by the C++ service layer. Complete

Phase 2 — iOS App

Qt/QML application targeting iOS 16+. Cupertino style. All ventocontrol features. Planned

Phase 3 — Android App

Same Qt/QML codebase, Material style. Requires a JNI bridge for WiFi multicast lock. Future

Phase 4 — SailfishOS App

Silica QML variant, packaged as an RPM for Harbour. The primary reason Qt was chosen. Future

Framework Rationale

The SailfishOS requirement is the deciding constraint. Its native UI toolkit is Silica — QML built on Qt Quick. No other serious cross-platform option (Flutter, React Native, KMP) has a credible SailfishOS path.

The protocol layer is written in Rust rather than C++. Rust's type system and ownership model make the binary protocol parsing correct by construction — no buffer overflows, no null-pointer errors, no uninitialized memory. The Rust crate has zero Qt dependencies and cross-compiles to all target triples from a single Cargo.toml.

Layer Model

Clean Architecture — source code dependencies point strictly inward. The inner layers know nothing about Qt, QML, or any UI framework.

QML UI Platform-adaptive pages and components. Cupertino on iOS, Material on Android, Silica on SailfishOS.
↓
ViewModel (C++) QObject subclasses with Q_PROPERTY + NOTIFY signals. No business logic.
↓
Service (C++) Owns FFI calls into the Rust client, the 2-second QTimer poll, and JSON persistence.
↓
FFI Bridge extern "C" functions exported from the Rust crate. Header generated by cbindgen.
↓
Protocol (Rust) Pure Rust. VentoClient, UDP sockets, binary packet builder/parser. Zero Qt dependencies. Compiled as staticlib.
The Qt/service/viewmodel layers are stubbed in CMakeLists.txt and will be activated in Phase 2. Only the Rust protocol crate and FFI bridge are implemented today.

FFI Bridge

The bridge uses a minimal, stable C ABI. All functions return a VentoStatus enum code; rich error text is available via vento_last_error(), which returns a thread-local pointer that lives until the next FFI call on that thread.

Error handling contract

// Every FFI function returns one of these.
typedef enum VentoStatus {
    Ok           = 0,
    ErrConnection = 1,  // UDP timeout, bind failure
    ErrChecksum  = 2,   // Packet checksum mismatch
    ErrProtocol  = 3,   // Unexpected response format
    ErrValue     = 4,   // Parameter out of range
    ErrDiscovery = 5,   // Broadcast discovery failure
    ErrUnsupported = 6, // Device does not support param
} VentoStatus;

const char *vento_last_error(void);  // human-readable detail

Client lifecycle

// Create — returns opaque pointer, never throws.
VentoClient *c = vento_client_new("192.168.1.50",
                                  "MYFAN00000000001",
                                  "1234");

// Use
VentoDeviceState state;
if (vento_get_state(c, &state) == Ok) { ... }

// Destroy
vento_client_free(c);

Optional fields in VentoDeviceState

Every optional sensor or configuration value in VentoDeviceState is accompanied by a _valid byte. The service layer must check the flag before reading the value. 

if (state.fan1_rpm_valid) {
    qDebug() << "Fan 1 RPM:" << state.fan1_rpm;
}

Build System

The project uses CMake with the Corrosion module (v0.5) to integrate the Rust crate into the C++ CMake build. Corrosion fetches itself via FetchContent, so no separate install is needed. 

cmake --preset desktop-debug   # Phase 1 local build
cmake --preset ios             # Phase 2 (requires macOS + Xcode)
cmake --preset android         # Phase 3 (requires NDK)
cmake --preset sailfish        # Phase 4 (requires Mer SDK)

The Rust crate uses cbindgen (a build dependency) to auto-generate protocol/include/protocol.h on every cargo build. The header is committed so CMake consumers do not need to run Cargo separately.

Source Files

Rust crate — protocol/src/

  • lib.rs Crate root. Re-exports all public items; declares all modules.
  • param.rs Protocol constants (CMD_PAGE, DEFAULT_PORT, etc.), the Func enum (Read/Write/WriteResp/Increment/Decrement/Response), the 49-variant Param enum with all parameter IDs, ParamMeta per-parameter metadata, and the WriteParam value type.
  • packet.rs Packet builder (build_read, build_write, build_write_resp, build_increment, build_decrement, build_discovery) and TLV parser (parse_response, parse_data_bytes). Handles CMD_PAGE boundary crossing, CMD_SIZE overrides, and CMD_NOT_SUP markers. All field decoders are here: decode_int, decode_ip, decode_text, decode_firmware, decode_rtc_time, decode_schedule, etc.
  • transport.rs VentoTransport — owns a configurable timeout duration. Three methods: send_recv (send + wait for one reply), send_only (fire and forget), discover (broadcast + collect multiple replies within the timeout window). Built on std::net::UdpSocket with no Qt dependency.
  • client.rs VentoClient — the primary public API. Provides get_state, power control, speed control, mode control, boost, timer, humidity, relay/voltage sensors, weekly schedule, RTC sync, filter, alarms, firmware, WiFi configuration, password change, cloud permission, factory reset, and fan discovery. Contains an internal DeviceStateBuilder that assembles a DeviceState from raw parameter maps.
  • models.rs Domain model structs: DeviceState (all fields Option<T>), FirmwareVersion, RtcTime, RtcCalendar, SchedulePeriod, TimerCountdown, FilterCountdown, MachineHours, WifiConfig, SetRtcTime, DiscoveredDevice.
  • error.rs VentoError enum with seven variants: Connection, Checksum, Protocol, Value, Discovery, UnsupportedParams, Io. Manual Display + Error implementations (no proc-macro dependencies). type Result<T> = std::result::Result<T, VentoError>.
  • ffi.rs All extern "C" exports. Defines the VentoStatus enum, the five C-compatible structs (VentoDeviceState, VentoSchedulePeriod, VentoRtcInput, VentoDiscoveredDevice, VentoDeviceList), thread-local vento_last_error(), client lifecycle functions, and all 27 operation functions.

Build & generated

  • protocol/build.rs Invokes cbindgen to regenerate include/protocol.h on every build.
  • protocol/include/protocol.h Generated C header. Contains all structs, the enum, and 27 function declarations. Include-guarded as ARABELLA_PROTOCOL_H.
  • protocol/examples/connect.rs Phase 1 Step 7 — Rust end-to-end validation tool. Runs against a fan or the Python simulator. Exercises get_state, turn_on, set_speed, set_manual_speed, set_ventilation, set_humidity_threshold, enable_weekly_schedule, set_schedule_period, get_schedule_period, turn_off, and validates rejection of out-of-range values.

Integration tests — tests/

  • tests/ffi_test.c Phase 1 Step 9 — C integration driver. Plain C program that links against the Rust staticlib and exercises 15 checks through the C ABI: client lifecycle, get_state, turn_on/off, set_speed, set_manual_speed, set_mode, set_humidity_threshold, weekly schedule, schedule period read/write, turn_off, 4 out-of-range rejections, and null-pointer safety. Exits 0 on full pass. Compile: gcc tests/ffi_test.c -I protocol/include protocol/target/debug/libarabella_protocol.a -lpthread -ldl -lm -o tests/ffi_test
  • tests/run_integration.sh Helper script: builds the Rust staticlib, compiles ffi_test.c, starts the Python fan simulator in the background, runs the driver against 127.0.0.1 SIMFAN0000000001 1111, stops the simulator, and reports the result. Flags: --no-rebuild (skip recompile), --keep-sim (leave simulator running).

Remaining Phases

The protocol crate (Phase 1) is complete and shared across all three platforms. Each platform phase adds a Qt/QML application layer on top of the same C FFI boundary. Platform-specific details are documented in the linked platform pages.

PhasePlatformStyleStatusDetails
2 iOS 16+ Cupertino Planned ios.html

Phase 3

Android — key constraints

Same QML codebase, Material style. Requires Android NDK, CHANGE_WIFI_MULTICAST_STATE permission, and a thin JNI bridge to acquire/release a WifiManager.MulticastLock on foreground/background transitions.

Phase 4

SailfishOS — key constraints

Silica QML variant, packaged as an RPM for the Harbour store. Requires the Mer SDK toolchain and Rust cross-compilation to armv7-unknown-linux-gnueabihf. The protocol crate needs no changes — it has no platform-specific code. The QML pages get Silica companion files (DashboardPage.sailfish.qml, etc.) selected at build time via a PLATFORM_SAILFISH define.
